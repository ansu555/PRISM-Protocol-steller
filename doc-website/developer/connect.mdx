# Connect a Client

PRISM Core is deployed on **Stellar Testnet**. To talk to it from a frontend, server script, or indexer you need three things:

1. A **Soroban RPC** server to simulate and submit transactions.
2. A **signer** — Freighter in the browser, or a `Keypair` on the server.
3. The **contract ID** for `prism_core`.

```
┌──────────────────┐     ┌──────────────────┐     ┌────────────────────┐
│ @stellar/         │────▶│ Contract +        │────▶│ prism_core          │
│ stellar-sdk       │     │ TransactionBuilder│     │ on Soroban testnet  │
│ (rpc.Server)      │     │ + signer          │     │ deposit / get_vault │
└──────────────────┘     └──────────────────┘     └────────────────────┘
```

Reads are dry-run simulations (no signature, no fee). Writes simulate, then prepare, sign, and submit.

## The client wrapper

The PRISM frontend wraps this stack in a small `ContractClient` (`app/lib/stellar.ts`) with two methods — `read` and `invoke`. You can copy the pattern or use the raw SDK directly.

```ts
import { rpc, Contract, TransactionBuilder, scValToNative } from '@stellar/stellar-sdk';

const server = new rpc.Server('https://soroban-testnet.stellar.org');
const contract = new Contract('CCULBBT4PA64GWXSKT4G7HOYQ4RXRNYY2JP5MZ2G73VKRFAJ6CHB3RZK');
```

## Steps

<Steps>
  <Step title="Provide a signer">
    **Server scripts** — load a `Keypair` from a secret seed (`S...`):

    ```ts
    import { Keypair } from '@stellar/stellar-sdk';
    const signer = Keypair.fromSecret(process.env.SECRET_SEED!);
    ```

    **Browser apps** — sign with [Freighter](https://www.freighter.app/) via the Stellar Wallets Kit. You never handle the user's secret; the wallet signs the prepared transaction:

    ```ts
    import {
      StellarWalletsKit, WalletNetwork, FREIGHTER_ID,
    } from '@creit.tech/stellar-wallets-kit';

    const kit = new StellarWalletsKit({
      network: WalletNetwork.TESTNET,
      selectedWalletId: FREIGHTER_ID,
    });
    const { address } = await kit.getAddress();
    ```
  </Step>

  <Step title="Read contract state">
    Reads are a single simulation — no signing, no fee. `get_vault`, `get_tranche`, `get_config`, and `get_loan` all return their value directly. Note many getters return an `Option`, decoded as the value or `null`:

    ```ts
    import {
      Contract, TransactionBuilder, BASE_FEE, Networks, nativeToScVal, scValToNative, rpc,
    } from '@stellar/stellar-sdk';

    async function getVault(vaultId: number) {
      const account = await server.getAccount(PUBLIC_SIM_ADDRESS);
      const tx = new TransactionBuilder(account, {
        fee: BASE_FEE,
        networkPassphrase: Networks.TESTNET,
      })
        .addOperation(contract.call('get_vault', nativeToScVal(vaultId, { type: 'u32' })))
        .setTimeout(30)
        .build();

      const sim = await server.simulateTransaction(tx);
      if (rpc.Api.isSimulationError(sim)) throw new Error(sim.error);
      return scValToNative(sim.result!.retval); // Vault | null
    }
    ```

    <Note>
      A read needs a source account only to build the simulation transaction — it is never signed or submitted. Any funded testnet address works (the frontend uses the deployed admin address).
    </Note>
  </Step>

  <Step title="Send a transaction">
    Writes follow **build → prepare → sign → send → poll**:

    ```ts
    async function sendTx(tx) {
      const prepared = await server.prepareTransaction(tx); // adds the Soroban footprint + resource fee
      prepared.sign(signer);                                 // or: await kit.signTransaction(prepared.toXDR())

      const sent = await server.sendTransaction(prepared);
      let res = await server.getTransaction(sent.hash);
      while (res.status === 'NOT_FOUND') {
        await new Promise((r) => setTimeout(r, 1500));
        res = await server.getTransaction(sent.hash);
      }
      if (res.status !== 'SUCCESS') throw new Error(`tx ${sent.hash} failed: ${res.status}`);
      return res;
    }
    ```

    <Note>
      `prepareTransaction` runs the simulation for you and attaches the resource footprint and fee. Always call it before signing a write — a raw `build()` transaction has no Soroban authorization data.
    </Note>
  </Step>
</Steps>
