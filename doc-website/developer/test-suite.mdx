# Test Suite

PRISM ships a Rust integration suite that exercises every contract function end-to-end using Soroban's `Env` test harness. Tests are not a checkbox — they are the primary defense against silent capital corruption in a multi-tranche credit system.

## Run it

<Steps>
  <Step title="Prerequisites">
    - Rust (stable)
    - The [Stellar CLI](https://developers.stellar.org/docs/tools/cli) for contract builds and live deploys
  </Step>

  <Step title="Run the suite">
    The Soroban test harness runs the contract in-process — no validator, no network, fully deterministic:

    ```bash
    cd soroban
    cargo test -p prism-core
    ```

    Narrow to a subset by substring, or show output:

    ```bash
    cargo test -p prism-core -- waterfall      # only waterfall tests
    cargo test -p prism-core -- --nocapture    # show println! output
    ```
  </Step>

  <Step title="Build the WASM">
    ```bash
    stellar contract build --package prism-core
    ```
  </Step>
</Steps>

## What the harness uses

Tests register the contract against a fresh `Env::default()` and drive it through real client calls:

```rust
#[test]
fn test_waterfall_locked_demo_numbers() {
    let env = Env::default();
    let contract_id = env.register(PrismCore, ());
    let client = PrismCoreClient::new(&env, &contract_id);
    // ... init, deposit, accrue_yield, assert NAV ...
    assert_eq!(prime_nav, expected_q64);
}
```

Token transfers use the Stellar Asset Contract test utilities, so `deposit` and `withdraw` move real balances inside the harness.

## Financial invariants

The suite proves four properties hold across the full deposit → accrue → loss → withdraw lifecycle:

| Invariant | Rule | Guarantees |
| --- | --- | --- |
| **Reserve invariant** | `contract_usdc_balance == Σ tranche.total_assets + loss_bucket_balance` | Accounting and physical USDC always agree |
| **Waterfall ordering** | Yield fills Prime, then Core; Alpha receives the residual | Income is distributed top-down, deterministically |
| **Loss cascade** | Losses absorb Alpha → Core → Prime | Subordination is enforced bottom-up |
| **NAV monotonicity** | NAV only moves on yield or loss | No phantom value created or destroyed |

NAV assertions use locked expected Q64.64 fixed-point values, so any drift in the math fails the build.

## Oracle attestation path

The collateral, Encrypt, and Cloak suites exercise Soroban's native ed25519 verification (`env.crypto().ed25519_verify`) and the fixed-layout attestation message format. The Rust verification in the contract and the TypeScript message builder in the client produce byte-identical input, proven by the suite passing. Replay protection is covered too: reusing an attestation nonce fails with `CollateralNonceReused`.

## Continuous integration

```yaml
name: Soroban Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          targets: wasm32-unknown-unknown
      - name: Test
        run: cd soroban && cargo test -p prism-core
```

No validator orchestration is required — `cargo test` runs the whole suite in-process. Release artifacts are built separately with `stellar contract build --package prism-core`.
