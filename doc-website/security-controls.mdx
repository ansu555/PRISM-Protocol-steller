# Security & Controls

PRISM financial logic must be deterministic, transparent, and enforceable on-chain.

## Access Control

PRISM separates privileged initialization from permissionless user flows.

Admin authority controls:

- Protocol initialization.
- Vault setup.
- Tranche setup.
- Emergency actions.
- Demo or fallback credit event triggers.

Users can permissionlessly:

- Deposit.
- Withdraw.
- Trade.
- Read protocol state.

## Pause Mechanism

The `prism_core` pause flag blocks:

- Deposits.
- Withdrawals.
- Admin pool seeding through `prism_core`.

While allowing:

- Accounting updates.
- Credit event resolution.
- Emergency reconciliation.

Direct Soroswap trading remains outside the vault contract's pause boundary. This matters because a vault pause should stop core state-changing user flows without pretending to control an external market.

## Determinism

All financial logic is:

- Transparent.
- Reproducible.
- Enforced on-chain.

Given the same vault state and input event, the protocol should produce the same output every time.

## Contract Authority

The `prism_core` contract is the on-chain authority. It controls:

- The vault USDC reserve.
- Tranche token mint and burn.
- The internal loss-accounting balance.
- Privileged state transitions.

Privileged flows still require configured admin authentication, but reserve transfers remain enforced by contract code.

## Checked Math

Credit accounting must avoid silent overflow, underflow, or precision surprises.

Core calculations should use:

- Checked arithmetic.
- Fixed-point math.
- Explicit rounding.
- Reverts on impossible states.

## Event Transparency

Every meaningful financial transition should be inspectable:

- Deposits.
- Withdrawals.
- Yield accrual.
- Credit events.
- Soroswap swaps.
- Tranche NAV updates.
